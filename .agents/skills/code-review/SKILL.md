# Code Review

When requested to review code changes, keep your review simple and concise. Focus on readability and obvious bugs.

As a best practice, try to rubberduck the code. Explaining the logic to a rubberduck often helps catch hidden issues early.

Remember: Keep it simple, and always rubberduck your code changes!

```
Title: [Component Name]: Brief Summary of Changes (e.g., Fixed Login Bug)

1. Overview
   - 1-2 sentences explaining what this change does.

2. Changes Made
   - Bullet points describing the key modifications.
   - Mention any significant logic changes.

3. Things to Watch Out For
   - Potential side effects or areas that need extra attention.
   - Any known limitations.

4. List any issues you found by priority:
   - P1: Critical issues that must be fixed.
   - P2: Important issues that should be fixed.
   - P3: Nice-to-have issues that can be fixed if time permits.
   - P4: Any other comments or suggestions.
```
